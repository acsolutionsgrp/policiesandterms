Overview
What is A2P 10DLC and why does it matter

A2P (Application-to-Person) 10DLC registration is required by US carriers for any business sending SMS messages via a 10-digit long code number. Without registration, messages will be filtered or blocked by carriers.

⚡
Critical Order of Operations Always create the A2P Brand FIRST. A Campaign cannot be submitted without an approved Brand. Brand approval must be confirmed before starting the Campaign form.
⚠️
Subaccounts need Brand Sharing For client subaccounts, the main AC Solutions Group account must first share its Brand. Go to main account → Regulatory Compliance → Brands → Share Brand → Enable on all subaccounts → Submit support ticket. Wait for Twilio confirmation before proceeding.

PART 2
Create the A2P Brand
Must be completed and approved before creating the Campaign

Log Into the Correct Twilio Subaccount

Each client has their own subaccount. Never create the Brand under the main AC Solutions Group account unless it's for AC Solutions Group itself.

  • Go to console.twilio.com
  • Click the account dropdown at the top of the page
  • Select the client's subaccount (e.g. NorthPark Medical Group)
  • Verify the account name shows in the top bar before proceeding

Navigate to Brand Registration

Messaging → Regulatory Compliance → A2P 10DLC Overview → Brands → Create A2P Brand

Alternatively, from Trust Hub: Registrations → A2P Brands → Create A2P Brand

Select Brand Type: Low Volume Standard

✅
Always select: Low Volume Standard This unlocks the "Low Volume Mixed" campaign use case at $1.50/month. If you accidentally select "Standard", the campaign will cost $10/month. This cannot easily be changed after submission.

Low Volume Standard is appropriate for the vast majority of clients. Only select "Standard" if the client sends extremely high message volumes (tens of thousands per day).

Submit and Wait for Approval

  • Review all fields — double-check EIN and website URL
  • Click Submit
  • Status will show Pending — approval is typically minutes to 24 hours
  • You will receive an email when approved
  • Do NOT create the Campaign until Brand status shows Approved
✅
North Park Medical Group's Brand was approved within minutes on June 3, 2026. Low Volume Standard brands typically approve quickly when EIN and website match.

Campaign Description and Sample Messages

Campaign Description — describe what the client will send, who receives it, and include opt-out language:

Campaign Description — North Park Medical Group This campaign sends appointment reminders, physical therapy scheduling updates, lab result notifications, referral coordination messages, and responses to patient inquiries from North Park Medical Group to patients who have provided consent during registration. Patients may opt out at any time by replying STOP. For help reply HELP. Message and data rates may apply. Message frequency varies based on appointment schedule.

This is the most scrutinized field and the #1 reason campaigns are rejected. Be explicit and specific. Include the exact verbal script if using Verbal opt-in.

Must include all of the following:

  • HELP instructions
  • "Msg & data rates may apply"
  • "Message frequency varies"
  • Privacy Policy URL
  • Terms URL

Policy Page System — GitHub + Cloudflare
How to generate, push, and host Privacy Policy, Terms, and SMS Opt-In pages for each client
🏗️
Architecture Overview All client policy pages live in one GitHub repository (acsolutionsgrp/policiesandterms) that auto-deploys to Cloudflare Pages on every push. Each client gets a subfolder. No DNS changes needed per client — the wildcard subdomain handles everything.

Example prompt to Claude:
"Generate privacy policy and terms pages for [Client Name]. Website: [URL]. Phone: [number]. Address: [address]. They send appointment reminders and billing notifications. Also generate an sms-optin page — keyword APPT to [phone]."

Claude will generate a downloadable ZIP. Extract it and copy the folders into ~/Documents/policiesandterms/[clientslug]/

⚠️
Do NOT commit ZIP files or .DS_Store files. Only commit the HTML index files. If accidentally committed, remove them: rm [clientslug]/*.zip [clientslug]/.DS_Store then git add + commit + push.

Step 3 — Push to GitHub

This triggers Cloudflare Pages to auto-deploy. Pages go live in ~60 seconds.

Terminal — Push to GitHub cd ~/Documents/policiesandterms git add . git commit -m "Add [Client Name] policy pages" git push
🔑
Git authentication uses a Personal Access Token. If prompted for a password, use your GitHub Personal Access Token (not your GitHub password). Token is embedded in the remote URL — if it expires, regenerate at github.com → Settings → Developer Settings → Personal Access Tokens → Tokens (classic) → repo scope.

Expected output when successful:

Success output Writing objects: 100% (X/X), X KiB | X MiB/s, done. remote: Resolving deltas: 100% (X/X), done. To https://github.com/acsolutionsgrp/policiesandterms.git abc123..def456 main -> main

Step 4 — Test All URLs in Browser

Wait 60 seconds after pushing, then test every URL before using them in Twilio:

URLs to verify (replace [clientslug]) https://policiesandterms.com/[clientslug]/privacy-policy ← Must load https://policiesandterms.com/[clientslug]/terms ← Must load https://policiesandterms.com/[clientslug]/sms-optin ← If Via Text
🚨
Never enter a URL in Twilio that you haven't tested in a browser first. A 404 or error page = instant campaign rejection.

Step 5 — Add Cloudflare Custom Domain (Optional)

Only needed if you want a branded subdomain like northparkmed.policiesandterms.com. The subfolder URLs work for Twilio without this step.

dash.cloudflare.com → Workers & Pages → policiesandterms → Custom Domains → Set up a custom domain

Enter: [clientslug].policiesandterms.com

Cloudflare will automatically add the DNS CNAME record since the domain is in the same account. Status will change from "Verification pending" to "Active" within a few minutes.

ℹ️
Wildcard DNS is already set up. A *.policiesandterms.com CNAME pointing to policiesandterms.pages.dev is already in Cloudflare DNS. However, Cloudflare Pages still requires each subdomain to be registered as a Custom Domain in the project settings — the DNS wildcard alone is not enough.

Common Rejection Reasons & Fixes
Learn from real rejections — Crescent Home Health Care was rejected for CTA issues
Rejection Reason Root Cause Fix
Issues verifying the Call to Action (CTA) Consent description too vague — reviewer couldn't verify how customers actually opt in Add the exact verbal script staff reads. For Via Text, add the hosted collateral URL. Be specific — don't say "verbally" without showing the exact words.
STOP/HELP missing from sample messages Sample messages didn't include opt-out and help instructions Add "Reply STOP to opt out, HELP for help" and "Msg & data rates may apply" to every single sample message — no exceptions.
Policy URLs not resolving Pages weren't deployed yet or had wrong URL Always test both URLs in a browser before submitting. Push to GitHub and wait 60 seconds for Cloudflare Pages to deploy.
Error: undefined on submit A required field is empty — usually consent description, message content checkboxes, or policy URLs Scroll back to top. Verify Embedded Links and Phone Numbers are checked. Check that consent description has content and both policy URLs are filled.
Subaccount missing shared brand Brand sharing ticket not submitted or not yet processed Main account → Regulatory Compliance → Brands → Share Brand → Enable on all subaccounts → Submit support ticket. Wait for Twilio to confirm (few hours to 1 day).
Brand rejected — identity mismatch EIN or website doesn't match TCR records for the business Verify EIN with client. Ensure website is live and matches the business name. Government/financial sites may block Twilio's verification crawlers — use a different IP if needed.

Final Submission Checklist

BEFORE

  • Client legal name, EIN, website, phone, address collected
  • Policy pages created in correct folder on Mac
  • git push successful
  • Both policy URLs tested and loading in browser
  • Correct Twilio subaccount selected
  • Brand Approved status confirmed
BRAND
A2P Brand Form
  • Low Volume Standard brand type selected
  • EIN entered correctly
  • Website URL is live and correct
  • Industry matches client's business
  • Submitted and confirmed Approved before proceeding
CAMPAIGN
A2P Campaign Form
  • Low Volume Mixed use case selected
  • Campaign description includes STOP/HELP/rates/frequency
  • Every sample message has STOP, HELP, and rates
  • Only correct opt-in methods checked
  • Consent field has exact verbal script and/or collateral URL
  • Embedded Links + Phone Numbers checked
  • Privacy Policy URL entered and tested
  • Terms URL entered and tested
  • Opt-out and Help fields left blank
AFTER
Post-Submission
  • $15 vetting fee confirmed
  • Campaign status shows In Review / In Progress
  • Monitor email for approval or rejection
  • If rejected: read reason, fix specific issue, resubmit
  • If approved: add phone numbers to Messaging Service Sender Pool
  • Test by sending a message from the registered number
